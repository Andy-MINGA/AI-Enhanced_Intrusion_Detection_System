## 🧭 What This Project Does 

Every day, computer networks are attacked. Tools like **Snort** monitor network traffic and raise an alarm whenever something suspicious is detected. The problem: Snort raises **way too many false alarms**. In our dataset, more than half of all alarms were noise. Security analysts waste most of their time chasing fake threats instead of real ones. This is called **alert fatigue**.

This project builds a **three-layer pipeline** that:

1. **Catches** all suspicious traffic with Snort; missing nothing
2. **Filters** the noise automatically using a machine learning model (XGBoost)
3. **Explains** each confirmed attack in plain English using the GPT API (GPT-4o mini), including what the threat is, how dangerous it is, and exactly what to do about it

The result: 237,240 noisy alerts reduced to 108,172 confirmed threats with 99% precision, and professional threat intelligence briefings produced for **$0.000603 total**.

## 🗂️ The Dataset

### Where the data comes from

The data was generated by the **Canadian Institute for Cybersecurity (CIC)** at the **University of New Brunswick, Canada** — the most-cited intrusion detection benchmark dataset in academic literature.

**How it was built:**  
The CIC constructed a complete laboratory network with 12 real computers running Windows, Ubuntu, and macOS. They simulated 25 realistic users browsing the web, sending email, and downloading files — to generate authentic normal traffic. Then their security researchers launched real cyberattacks from a Kali Linux machine at documented times, recording every single network packet.

Because the attack times were precisely documented, every network flow could be labelled: either **BENIGN** (normal traffic) or a specific attack type. This is what makes it a *labelled* dataset — essential for training and evaluating machine learning models.

### Which file we used

The dataset comes in two distributions. We use the **GeneratedLabelledFlows** version — specifically the file:

```
Wednesday-workingHours.pcap_ISCX.csv
```

This file contains **692,703 labelled network flows** with full IP address information, which is required for our matching strategy. The alternative `MachineLearningCSV` distribution contains only statistical features with no IP addresses, making it incompatible with our pipeline.

### Download the dataset

1. Go to: [https://www.unb.ca/cic/datasets/ids-2017.html](https://www.unb.ca/cic/datasets/ids-2017.html)
2. Download the **Wednesday** PCAP file (approximately 13 GB)
3. Download the **GeneratedLabelledFlows** CSV archive
4. Extract `Wednesday-workingHours.pcap_ISCX.csv`
5. Place the PCAP file at `D:\PCAP\wednesday.pcap` (or update the paths in the notebooks)

## 🚀 Pipeline Walkthrough

### Phase 1 — Snort Alert Generation & Ground Truth Matching

**What happens here:**  
The 13 GB PCAP file is too large to process all at once, so it is split into 14 chunks of approximately 1 GB each. Snort 2.9 processes each chunk and generates alerts — one row per suspicious event. All 14 output files are merged into a single master dataset of 237,240 alerts.

**The column-shift problem:**  
Snort has a bug where it silently omits empty fields in its CSV output instead of writing an empty placeholder. This causes all subsequent columns in a row to shift left. A custom `fix_row()` function detects and corrects these shifts by anchoring on the protocol field, which is always expected at column index 8.

**Ground truth matching:**  
To know whether each Snort alert is a real attack or a false alarm, every alert is matched against the ISCX ground truth file using a three-tier strategy:

| Tier | Match Type | Logic |
|------|-----------|-------|
| 1 | Forward match | Source IP + Destination IP + Destination Port |
| 2 | Reversed match | Handles bidirectional flows |
| 3 | NAT fallback | Handles post-firewall address translation |

**Snort baseline:**
- 107,107 True Positives (real attacks correctly flagged)
- 130,133 False Positives (noise — 54.9% of all alerts)
- Precision: 0.4515 · Recall: 1.0000

---

### Phase 2 — XGBoost False-Positive Suppression

**What XGBoost does:**  
XGBoost is a machine learning algorithm that builds 200 decision trees sequentially, where each tree learns from the mistakes of the previous ones. It is trained on the 237,240 labelled alerts to learn the difference between real attacks and noise.

Instead of giving a binary yes/no answer, XGBoost assigns a **confidence score** between 0 and 1 to each alert. A score of 0.97 means 97% confident this is a real attack. A score of 0.03 means it is probably noise.

**Feature engineering:**  
From 31 original Snort columns, 12 features are kept. IP addresses are deliberately excluded — if the model learned that 172.16.0.1 is an attacker, it would be useless on any other network. We keep packet-level features like port numbers, TCP flags, and packet lengths.

**The threshold decision:**  
The threshold is the cutoff that converts XGBoost's confidence score into a final decision:
- Score ≥ threshold → **pass to analyst** (predicted attack)
- Score < threshold → **suppress** (predicted noise)

At the default threshold of 0.50, XGBoost was silently discarding about 85,000 real attacks because it was only 30–40% confident about them. By lowering the threshold to **0.01** — passing anything with even a 1% chance of being an attack — missed attacks dropped from ~160 to just 3.

| Threshold | Missed Attacks (FN) | Noise Admitted (FP) |
|-----------|-------------------|-------------------|
| 0.50 | 32 | 10 |
| 0.10 | 17 | 81 |
| **0.01** | **3** | **245** |

**Rule whitelist:**  
One Snort rule — *"TCP Timestamp is outside of PAWS window"* — has 99.8% precision with only 3 false positives. It is hard-whitelisted: always passed through regardless of XGBoost's score, eliminating any risk of suppressing its alerts.

**Final XGBoost results (full dataset):**
- False Positives: 130,133 → **1,068** (−99.2%)
- Missed Attacks: 1 → **3** (negligible)
- Precision: 0.45 → **0.99**
- Recall: remains **1.00**

### Phase 3 — LLM Threat Triage (GPT-4o mini)

**Why XGBoost is not enough:**  
After filtering, the analyst still receives 108,172 rows of raw data like this:

```
2017-07-05 10:47:32 | Reset outside window |
172.16.0.1 → 192.168.10.50 | Port: 80 | TCP | Flags: 0x04
```

XGBoost confirmed this is a real attack. But it cannot say: *what kind? how dangerous? what CVE? what should we do?* XGBoost is a sorting machine — it only knows numbers.

**What the LLM does:**  
The 108,172 confirmed attacks are grouped by attack type. One representative alert per group is sent to the OpenAI GPT-4o mini API. The model returns a structured JSON briefing for each attack type.

**API cost:**  
The entire LLM phase used 4,018 tokens and cost **$0.000603** — less than one tenth of one cent.

**LLM output summary:**

| Attack Type | Alerts | Severity | CVE |
|-------------|--------|----------|-----|
| DoS Hulk | 105,131 | Critical | None |
| DoS GoldenEye | 1,558 | Critical | None |
| DoS slowloris | 279 | Critical | None |
| DoS Slowhttptest | 1,199 | High | None |
| **Heartbleed** | **5** | **Critical** | **CVE-2014-0160** |

**Heartbleed** is the most significant finding. It is the only named software vulnerability in this dataset — a flaw in OpenSSL's heartbeat protocol that allows an attacker to read up to 64 KB of server memory per request, potentially exposing private keys and passwords. The LLM correctly identified CVE-2014-0160 and recommended patching OpenSSL to ≥1.0.1g and revoking all SSL certificates.

**Output files:**
- `llm_threat_briefings.json` — structured JSON, one entry per attack type
- `llm_threat_report.txt` — human-readable text report

## 🔮 Next Step — GPT-2 Fine-Tuning

**Why this matters:**  
GPT-4o mini works by calling OpenAI's external servers — a commercial, paid API. The final research objective is to instead fine-tune **GPT-2**, a smaller open-source language model whose weights and architecture are publicly available, on our own cybersecurity data.

**What fine-tuning means:**  
Fine-tuning is like giving a knowledgeable generalist a cybersecurity internship. Before fine-tuning, GPT-2 knows general English but is vague about security topics. After training on CVE reports, Snort documentation, and our confirmed attack alerts, it learns to speak the language of cybersecurity precisely.

**Method:** [LoRA (Low-Rank Adaptation)](https://arxiv.org/abs/2106.09685) — a technique that trains only a small fraction of the model's parameters, making fine-tuning feasible on a single free Google Colab GPU in approximately 2 hours.

**Training data plan:**
- CVE descriptions from the [National Vulnerability Database (NVD)](https://nvd.nist.gov) — free, public
- The 108,172 confirmed TP alerts from this pipeline
- Snort rule documentation
- OWASP attack descriptions

**Evaluation:**
- Perplexity score before vs after fine-tuning
- Side-by-side output comparison: fine-tuned GPT-2 vs GPT-4o mini on identical alerts
- CVE accuracy, remediation quality, JSON structure reliability

**Research question:**  
*"How much does domain fine-tuning of a small open-source model close the capability gap with a large commercial model for cyber risk assessment?"*

## 🔑 Key Concepts Explained Simply

| Term | Plain English |
|------|--------------|
| **PCAP file** | A recording of every packet sent on a network — like a DVR for internet traffic |
| **IDS** | Intrusion Detection System — software that monitors network traffic for attacks |
| **Snort** | The most widely used open-source IDS in the world |
| **False Positive** | An alarm that fires but turns out to be nothing — a false alarm |
| **False Negative** | A real attack that was missed and no alarm fired |
| **Precision** | Of all alarms raised, what fraction were real attacks? |
| **Recall** | Of all real attacks happening, what fraction did the system catch? |
| **XGBoost** | A machine learning algorithm that builds many decision trees to classify data |
| **Threshold** | The confidence cutoff — below this, suppress the alert; above it, pass it through |
| **LLM** | Large Language Model — an AI that understands and generates human language |
| **CVE** | Common Vulnerabilities and Exposures — a universal ID number for known security flaws |
| **Fine-tuning** | Taking a pre-trained AI model and training it further on your own specific data |
| **LoRA** | A method for fine-tuning that only updates a small fraction of model parameters |
| **NAT** | Network Address Translation — a firewall technique that changes IP addresses |
| **Alert fatigue** | When analysts stop paying attention because there are too many false alarms |
